## 1. UI-фреймворк для оболочки

**Вопрос:** Какой подход к построению Shell App предпочтителен?

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Vanilla JS** | Нет зависимостей, максимальный контроль, простота деплоя | Больше boilerplate, нет компонентной модели из коробки |
| **Lit (Web Components)** | Лёгкий (~5KB), нативные Web Components, будущее стандартов | Менее популярен, меньше готовых компонентов |
| **Vue 3** | Богатая экосистема, реактивность, компоненты | Зависимость от фреймворка, тяжелее |

> **Решение: Vanilla JS + Web Components (Custom Elements).**
>
> Каждый микросервис оформляется как Custom Element (например `<service-card-converter>`).
> Не требует сборки (Webpack/Vite) на начальном этапе. Новые сервисы подключаются
> добавлением `<script src=".../remote-entry.js">`. Консистентно с существующим
> `spec-converterv2`, который уже использует Vanilla JS.

## 2. Standalone vs Embedded режим сервисов

**Вопрос:** Должны ли микросервисы сохранять возможность работать автономно (без оболочки)?

> **Решение: Вариант B — Dual mode через Custom Elements.**
>
> - Фронтенд микросервиса переписывается в виде JS-класса (Web Component).
> - `index.html` внутри сервиса сохраняется для автономного использования и тестирования.
> - Оболочка (Shell) импортирует JS-файл компонента и рендерит его в сетке.
> - Shadow DOM обеспечивает изоляцию стилей между сервисами.

## 3. Drag-and-drop: на уровне карточки или внутри карточки?

**Вопрос:** Как должен работать drag-and-drop в оболочке?

> **Решение: Вариант C — Комбинация.**
>
> - **Quick mode:** файл перетаскивается на карточку в сетке → обработка
>   с дефолтными настройками, прогресс и результат отображаются на карточке.
> - **Advanced mode:** клик по заголовку карточки → раскрытие (модалка / side-panel)
>   с логами, выбором провайдера (Anthropic/OpenRouter) и предпросмотром.

## 4. Маршрутизация API

**Вопрос:** Какую схему маршрутизации использовать?

> **Решение: Вариант A — Path-based через Nginx Reverse Proxy.**
>
> - `localhost/` → Shell App (статика)
> - `localhost/api/spec-converter/` → Flask :5001
> - `localhost/api/another-tool/` → сервис :5002
>
> Избавляет от CORS-проблем, упрощает деплой, не требует DNS-настройки.

## 5. Какие будущие сервисы планируются?

**Вопрос:** Есть ли уже идеи для второго и последующих микросервисов?

> **Решение: Заложены 4 направления.**
>
> | Сервис | Категория |
> |--------|-----------|
> | Image-to-Text (OCR) — извлечение текста из фото штампов чертежей | Конвертеры |
> | DOCX Merger — сборка ПЗ из кусков по шаблону | Генераторы документов |
> | Project Validator — проверка соответствия шифров в именах и внутри PDF | Проверка / Нормоконтроль |
> | (spec-converterv2 — существующий) | Конвертеры |
>
> **Категории сайдбара:** Конвертеры, Проверка/Нормоконтроль, Генераторы документов.

## 6. Аутентификация

**Вопрос:** Нужна ли аутентификация пользователей?

> **Решение: Отложить (Zero-auth).**
>
> На текущем этапе — без аутентификации. Если понадобится ограничить доступ,
> добавляется Basic Auth на уровне Nginx без изменений в коде микросервисов.
> Сценарий использования: локальный / командный инструмент.

## 7. Хранение и история операций

**Вопрос:** Нужно ли хранить историю конвертаций и обработанные файлы?

> **Решение: Stateless + LocalStorage.**
>
> - Последние 10 операций хранятся в `localStorage` браузера (имя файла, дата, статус).
> - Обработанные файлы удаляются сразу после выдачи (или через 1 час по cron).
> - Без базы данных на текущем этапе.

## 8. Развёртывание

**Вопрос:** Где будет развёрнута платформа?

> **Решение: Docker Compose.**
>
> `docker-compose.yml` поднимает:
> - 1 контейнер Nginx (Shell App + Gateway)
> - 1 контейнер spec-converterv2 (Flask)
> - N контейнеров для будущих сервисов
>
> Запуск одной командой: `docker-compose up`. Подходит для локальной машины
> и командного сервера.

## 9. Управление API-ключами

**Вопрос:** Как управлять API-ключами для Vision провайдеров?

> **Решение: `.env` файлы на уровне сервисов.**
>
> - Каждый микросервис управляет своими ключами через `.env`.
> - В репозитории — `.env.example` (без секретов).
> - Опционально: экран настроек в оболочке для смены ключей через API
>   без перезагрузки Docker.

## 10. Обработка ошибок и повторные попытки

**Вопрос:** Какая стратегия при ошибках Vision API?

> **Решение: Прозрачная индикация (Badges).**
>
> - Vision-fallback → иконка 👁 на карточке.
> - Ошибка → карточка окрашивается красным, в Advanced view — Python-трейсбек.
> - Кнопка «Только Vision» для принудительного выбора метода.
> - Retry: 3 попытки с экспоненциальной задержкой (текущая логика сохраняется).

## 11. Масштаб и ресурсы

**Вопрос:** Сколько пользователей одновременно будут работать с платформой?

> **Решение: Flask + Gunicorn (синхронно) для начала.**
>
> - Начальный сценарий: 1-5 пользователей.
> - Следующий шаг: Task Queue (Celery + Redis) — пользователь загружает файл,
>   получает `task_id`, Shell App поллит статус готовности.
> - Gunicorn с 2-4 воркерами достаточен для начала.

## 12. Тестирование

**Вопрос:** Какой уровень тестового покрытия требуется?

> **Решение: Snapshot Testing на промежуточном JSON.**
>
> - Папка `gold_standard/` с парами `input.pdf` / `expected.json`.
> - Тест сравнивает промежуточный JSON (выход `pdf_text_extractor`), а не Excel.
> - Порог прохождения: совпадение > 95%.
> - Unit-тесты для `fix_encoding()`, `find_column_mapping()`, `normalize_table_to_9cols()`.
> - E2E-тесты — на этапе интеграции в оболочку.
